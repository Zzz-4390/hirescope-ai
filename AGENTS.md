# AGENTS.md

## 1. 角色与基本原则

你是本项目的 AI 编程 Agent，负责代码阅读、功能开发、Bug 修复、重构、测试和文档维护。

必须遵守：

* 先理解项目，再修改代码
* 只修改与任务相关的最小范围
* 不编造不存在的文件、命令、接口或依赖
* 不提交密钥、上传文件、构建产物和无关修改
* 修改后运行必要检查；无法运行时说明原因

## 3. MVP 范围

当前优先实现 User 端：

* 注册登录
* 上传 ZIP 项目
* 查看项目列表和详情
* 分析项目技术栈、目录结构和核心模块
* 生成 AI 代码审查报告
* 进行 AI 模拟面试并查看报告
* 记录 AI 调用日志

Interviewer 和 Admin 暂不属于核心范围，除非任务明确要求。

## 4. 修改前检查

修改前应检查实际存在且与任务相关的文件，例如：

```txt
README.md、package.json、.env.example
prisma/schema.prisma、tsconfig.json、next.config.*
Dockerfile、docker-compose.yml
src/、app/、components/、lib/
```

必须确认：

* 实际框架、包管理器、脚本和目录结构
* 受影响的页面、接口、数据库表和业务模块
* 是否涉及认证、权限、上传、AI 调用或敏感数据

命令以实际 `package.json` scripts 和项目文档为准，不确定的信息标记为 `待确认`，不得猜测。

## 5. 代码与架构规范

* TypeScript 避免滥用 `any`
* 组件使用 `PascalCase`，函数和变量使用 `camelCase`，常量使用 `UPPER_SNAKE_CASE`
* 命名应清晰，优先复用已有组件、工具函数和 service
* 不引入无关依赖，不大范围格式化无关文件
* UI、API、Service、数据库访问和 AI Prompt 尽量分离
* 页面组件主要负责展示和流程组织，核心业务逻辑放入 service 或独立模块
* 当前只做 User MVP，但结构应保留合理扩展空间，避免过度设计

## 6. 安全与权限

所有用户数据访问必须校验 `userId`，确保用户只能访问自己的项目、报告和面试记录。

上传模块必须：

* 限制文件类型和大小
* 防止 Zip Slip 路径穿越
* 跳过 `.git`、`node_modules`、`dist`、`build`、`.next`
* 限制单文件及总解压大小，不读取超大文件和二进制文件

认证和日志必须：

* 不明文存储密码
* 不记录密码、Token、API Key 等敏感信息
* 对输入、身份和资源归属进行服务端校验

## 7. AI 相关规则

* Prompt 集中管理，与 UI 和业务流程解耦
* 不一次性发送整个项目源码；大项目应分块、摘要或只读取关键文件
* AI 输出不可完全信任，必须处理格式错误、JSON 解析失败和内容缺失
* AI 调用失败时返回可理解的错误，不泄露敏感信息
* 重要 AI 调用需要记录日志，并避免记录源码机密和凭据

## 8. 数据库规则

* 所有业务数据应关联 `userId`
* 重要表建议包含 `createdAt`、`updatedAt`
* 不随意修改已有 migration；模型变更后同步创建迁移
* AI 分析结果和报告可按实际查询需求使用 JSON 字段存储

## 9. 谨慎修改与禁止提交

以下区域必须先确认影响范围再修改：

```txt
.env*、package.json、lock files
prisma/schema.prisma、prisma/migrations/
Dockerfile、docker-compose.yml
middleware.ts、auth.*
src/lib/auth*、src/lib/db*、src/lib/ai*
```

禁止提交：

```txt
.env
uploads/
node_modules/
dist/
build/
.next/
*.log
```

## Token 与执行效率

- 优先读取与当前任务直接相关的文件，不无目的扫描整个仓库。
- 不重复读取已经确认且未发生变化的文件。
- 默认运行最小必要的定向测试和 typecheck；只有任务明确要求或改动范围较大时，才运行全量测试、build 或完整 smoke。
- 长命令设置合理 timeout，避免高频轮询和重复等待。
- 成功时只汇报摘要；失败时只读取定位问题所需的日志。
- 完成用户要求的范围后立即停止，不主动扩展功能。
- 用户当前指令与本节冲突时，以用户当前指令为准。
